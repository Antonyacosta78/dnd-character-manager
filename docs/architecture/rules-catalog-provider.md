# Architecture: Rules Catalog Provider

## Purpose

This document defines the runtime contract for reading rules content and the implementation strategy for swapping data-access engines without changing application/domain logic.

It is intentionally explicit because this contract is the boundary between:

- gameplay logic that must remain stable and testable, and
- evolving data ingestion/runtime strategies (`derived` now, `raw` later).

## Design Decision

Use a single abstraction named `RulesCatalog`.

- No additional facade layer for the same concern.
- Dependency injection is used only to bind implementation at composition time.
- Domain/application code depends on `RulesCatalog` only.

This avoids abstraction duplication (`RulesFacade` + `RulesCatalogProvider`) while preserving swapability.

## Contract Goals

The `RulesCatalog` contract must satisfy:

1. source-format agnosticism (no `_copy`, `_mod`, or parser internals in API surface)
2. deterministic results for a fixed dataset fingerprint
3. sufficient query capabilities for character creation/progression flows
4. explicit dataset lineage exposure for observability and safety checks
5. additive extensibility when new gameplay areas are implemented

## Type-Level Contract

Reference types:

```ts
type RulesRef = {
  name: string;
  source: string;
};

type DatasetVersion = {
  provider: "derived" | "raw";
  fingerprint: string;
  label?: string;
  generatedAt?: string;
};
```

Primary interface (aggregated, namespaced shape):

```ts
interface RulesEntityReader<TEntity, TFilter = RulesFilter> {
  get(ref: RulesRef): Promise<TEntity | null>;
  list(filter?: TFilter): Promise<TEntity[]>;
}

interface SubclassReader {
  get(args: { classRef: RulesRef; subclassRef: RulesRef }): Promise<SubclassDefinition | null>;
  listByClass(classRef: RulesRef): Promise<SubclassDefinition[]>;
}

interface FeatureReader {
  resolve(ref: FeatureRef): Promise<FeatureDefinition | null>;
}

interface RulesCatalog {
  getDatasetVersion(): Promise<DatasetVersion>;

  classes: RulesEntityReader<ClassDefinition>;
  subclasses: SubclassReader;
  races: RulesEntityReader<RaceDefinition>;
  backgrounds: RulesEntityReader<BackgroundDefinition>;
  spells: RulesEntityReader<SpellDefinition, SpellFilter>;

  // This namespace is added when feat support becomes required.
  feats: RulesEntityReader<FeatDefinition, FeatFilter>;

  features: FeatureReader;
}
```

Contract semantics:

- `*.get(...)` methods return `null` for not found, not transport-level errors.
- methods must be deterministic for identical input and dataset fingerprint.
- returned entities must match domain expectations, not raw source shape.

### Why namespaced readers instead of flat methods

The namespaced shape (`rulesCatalog.classes.get`, `rulesCatalog.feats.list`) is preferred over flat methods (`getClass`, `listFeats`) for this project because:

- it scales cleanly as entities expand
- it avoids method-name explosion and keeps discoverability high
- it lets each namespace have domain-specific extras (`subclasses.listByClass`)

This is still a single abstraction; it is not an additional facade layer.

## Extensibility Policy

The catalog contract is designed for additive extension.

When a feature area requires new rules content (for example, feats), the expected change is:

1. add new namespace to `RulesCatalog` (for example, `feats`)
2. implement `get`/`list` semantics in all active providers
3. extend contract and parity test suites
4. gate feature rollout on provider support

For the concrete feats case, adding support means introducing:

- `rulesCatalog.feats.get(ref)`
- `rulesCatalog.feats.list(filter?)`

Breaking changes to existing namespaces are avoided; contract evolution should be additive until a deliberate major-version migration.

## Error Taxonomy

Implementations should map internal failures into stable error categories:

- `RulesCatalogNotFoundError` - semantic not found where exception semantics are required
- `RulesCatalogUnavailableError` - provider or backing store unavailable
- `RulesCatalogDatasetMismatchError` - integrity/lineage mismatch with expected active dataset
- `RulesCatalogContractViolationError` - adapter emitted malformed contract data

Not-found-as-null remains preferred for read methods; exceptions are for operational/consistency failures.

## Implementations

### `DerivedRulesCatalog` (default)

Characteristics:

- reads from normalized catalog storage generated by ingestion pipeline
- optimized for predictable request-path behavior
- simpler operational profile for v1

Internal composition recommendation:

- `ClassCatalogReader`
- `SubclassCatalogReader`
- `RaceCatalogReader`
- `BackgroundCatalogReader`
- `SpellCatalogReader`
- `FeatCatalogReader`
- `FeatureCatalogReader`

`DerivedRulesCatalog` aggregates these readers into the namespaced `RulesCatalog` interface.

Use when:

- delivery speed and runtime stability are the priority
- source format changes should be isolated to ingest/normalize pipeline

### `RawRulesCatalog` (future)

Characteristics:

- resolves from source input semantics at runtime
- may reduce derived-data duplication
- higher runtime complexity and stricter performance/consistency burden

Internal composition recommendation mirrors derived provider, but with resolver-oriented implementations per namespace.

This is likely what you referred to as separate transformer classes. The pattern is correct, but keep them behind the same aggregated contract.

Use when:

- parity with derived provider is proven
- operational profile is acceptable under production-like load

## Dependency Injection and Composition

DI is intentionally lightweight and localized to composition root.

Recommended wiring shape:

```ts
export function createRulesCatalog(config: AppConfig): RulesCatalog {
  switch (config.rulesProvider) {
    case "raw":
      return new RawRulesCatalog({config});
    case "derived":
    default:
      return new DerivedRulesCatalog({config});
  }
}
```

Derived provider composition sketch:

```ts
export function createDerivedRulesCatalog(deps: DerivedDeps): RulesCatalog {
  return {
    getDatasetVersion: () => deps.versionRepo.getActive(),
    classes: createClassCatalogReader(deps),
    subclasses: createSubclassCatalogReader(deps),
    races: createRaceCatalogReader(deps),
    backgrounds: createBackgroundCatalogReader(deps),
    spells: createSpellCatalogReader(deps),
    feats: createFeatCatalogReader(deps),
    features: createFeatureCatalogReader(deps),
  };
}
```

Rules:

- application services receive `RulesCatalog` via constructor/function injection
- route handlers/server actions consume pre-wired services
- no dynamic import branching in random call sites
- per-namespace reader construction belongs in provider factory code, not in application services

## Caching and Performance Expectations

Contract-level expectations:

- `list*` endpoints used for UI option sets should be cacheable by dataset fingerprint
- single-entity lookups should support stable low-latency access
- provider-level caches must be invalidated when active dataset fingerprint changes

The cache strategy can differ by provider as long as behavior remains contract-consistent.

## Integrity and Lineage Requirements

Each provider must return trustworthy version metadata via `getDatasetVersion()`.

Runtime checks should ensure:

- expected active fingerprint == provider fingerprint
- provider identity is known (`derived` or `raw`)
- mismatch handling follows integrity mode policy

Integrity mode policy:

- `strict`: fail closed
- `warn`: continue with structured warnings
- `off`: bypass for controlled local experimentation

## Contract Testing Requirements

Required tests:

- shared contract suite that runs against every implementation
- fixture-driven deterministic tests for representative entity types
- behavioral parity tests (`DerivedRulesCatalog` vs `RawRulesCatalog`)
- resilience tests for unavailable/mismatch states
- namespace coverage tests to ensure newly added domains (for example `feats`) are fully implemented

The contract suite is the gatekeeper for provider swaps.

## Provider Migration Protocol

Migration from `derived` default to `raw` default requires:

1. contract suite green on both providers
2. parity threshold achieved on agreed fixture corpus
3. shadow-read comparison run in staging/preview
4. documented rollback plan by config flip

Additional requirement for newly added namespaces:

- do not mark namespace production-ready until both providers either implement it or feature gating explicitly restricts to supported provider(s).

Do not switch default provider based on qualitative confidence alone.

## Non-Goals

This document does not define:

- source sync/extract mechanics (see parsing pipeline)
- normalization algorithms
- UI-level data presentation decisions
- write/edit workflows for source input
